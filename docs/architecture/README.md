# Архитектура репозитория

## Цель структуры

Репозиторий переведён из плоского layout в модульный монорепозиторий, чтобы сохранить текущую рабочую API+ML связку и при этом безболезненно добавить новые приложения рядом с существующим API.

## Текущие зоны ответственности

- `apps/api` содержит только API-слой и контейнеризацию API.
- `ml/model` содержит обучение, inference, explainability и поддержку данных.
- `ml/artifacts` хранит бинарные артефакты модели отдельно от кода.
- `ml/data/raw` и `ml/reports` остаются рядом с ML-кодом как operational outputs.
- корневые `api.py` и `main.py` сохранены как совместимые entrypoint-обёртки.

## Принципы роста

- Новый пользовательский модуль добавляется в `apps/<module>`.
- Отдельные backend-сервисы можно позже добавить в `services/`, когда они реально появятся.
- Общие библиотеки выносятся в `shared/` только после появления повторного использования; сейчас это избыточно.
- Документация растёт по разделам в `docs/`, а не в одном корневом файле.
